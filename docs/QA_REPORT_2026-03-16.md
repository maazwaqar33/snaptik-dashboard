## P0 — Critical

### SEC-015: NoSQL Injection via Operator Object in Login Body
**Endpoint:** `POST /api/v1/auth/login`
**Impact:** Server crash (500) + potential authentication bypass
**Reproduced:** Yes — `{"email":{"$gt":""},"password":"x"}` → 500

**Root Cause:**
`auth.controller.ts` calls `email.toLowerCase()` directly on the request body value without first validating that it is a string. When `email` is a MongoDB query operator object, `.toLowerCase()` throws a `TypeError` that propagates uncaught to Express's default error handler.

**Reproduction:**
```bash
curl -s -X POST http://localhost:5001/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":{"$gt":""},"password":"x"}'
# → HTTP 500 Internal Server Error
```

**Fix:**
```typescript
// auth.controller.ts — before any usage of email
if (typeof email !== 'string') {
  return res.status(400).json({ message: 'Invalid credentials' });
}
```
Or add a global sanitise middleware (`express-mongo-sanitize`) that strips `$` keys from body/query/params.

## P1 — High

### BUG-001: Oversized Body Returns 500 Instead of 413
**Endpoint:** Any — e.g. `POST /api/v1/auth/login`, `PATCH /api/v1/settings/:key`
**Impact:** Uncaught `PayloadTooLargeError` crashes the request; no client-friendly error
**Reproduced:** Yes — body > 2 MB → 500 (automated test confirms)

**Root Cause:**
`index.ts` sets `express.json({ limit: '2mb' })`, which causes Express to emit a `PayloadTooLargeError` on oversized bodies. This error type is not handled in the global error handler, so Express falls through to its default 500 response.

**Fix:**
```typescript
// In global error handler (index.ts)
app.use((err: Error, _req: Request, res: Response, _next: NextFunction) => {
  if (err.type === 'entity.too.large') {
    return res.status(413).json({ message: 'Request body too large' });
  }
  // ... existing handler
});
```

### BUG-002: Mongoose CastError on Invalid ObjectId → 500
**Endpoints:** `GET /api/v1/tickets/:id`, `PATCH /api/v1/tickets/:id`, `DELETE /api/v1/tickets/:id`
**Impact:** Any URL with a non-ObjectId segment crashes the server
**Reproduced:** Yes — `/api/v1/tickets/not-an-objectid` → 500

**Root Cause:**
Mongoose throws `CastError` when `findById('not-an-objectid')` is called with a non-24-hex string. This error is not caught in the ticket controller or the global error handler.

**Fix (Option A — controller level):**
```typescript
import { isValidObjectId } from 'mongoose';
// At top of getTicket, updateTicket, deleteTicket:
if (!isValidObjectId(req.params.id)) {
  return res.status(400).json({ message: 'Invalid ticket ID' });
}
```

**Fix (Option B — global error handler):**
```typescript
import { Error as MongooseError } from 'mongoose';
if (err instanceof MongooseError.CastError) {
  return res.status(400).json({ message: 'Invalid ID format' });
}
```

**Note:** `DELETE /api/v1/tickets/not-an-objectid` returns 404 (not 500) — the DELETE route does not exist at all, which is a separate gap.

### BUG-003: Missing Core API Endpoints (Frontend Falls Back to Stub Data)
**Endpoints (confirmed 404):**
- `GET /api/v1/analytics/dashboard` — Analytics page uses in-memory stubs
- `GET /api/v1/content/moderation-queue` — Content page uses stubs
- `GET /api/v1/reports/kanban` — Reports page uses stubs
- `POST /api/v1/content/:id/approve` — Approve action is non-functional
- `POST /api/v1/content/:id/remove` — Remove action is non-functional
- `POST /api/v1/content/:id/defer` — Defer action is non-functional

**Impact:** All moderation actions are simulated; no data persists. The UI correctly shows an orange warning banner noting the API fallback, but this is not production-ready.

**Fix:** Implement the missing route handlers. The frontend contract is already defined; the API routes need to be wired to real Mongoose models.

## P2 — Medium

### BUG-004: No `POST /api/v1/tickets` Route (Ticket Creation)
**Impact:** Support tickets can only be created externally (e.g., mobile app). Admins cannot open tickets from the dashboard.
**Reproduced:** `POST /api/v1/tickets` → 404

### BUG-005: Login Form Shows No Error on Invalid Credentials
**Location:** `/login` page, `apps/web/`
**Impact:** When login fails (wrong email/password), the form silently resets with no error message or toast. The user has no feedback.
**Reproduced:** Yes (E2E screenshot `e2e-02-login-invalid.png`)
**Fix:** Catch the auth error in the login form's submit handler and display an inline error or toast.

### BUG-006: Ban User Button Visually Enabled but Accessibility Tree Says `[disabled]`
**Location:** Users page → Ban User modal
**Impact:** The button is styled as an active red button but the DOM attribute `disabled` prevents clicking. Misleading UI — a user could spend time clicking a disabled button.
**Reproduced:** Yes (E2E screenshot `e2e-05-ban-modal.png`)
**Fix:** Either remove `disabled` if the button should work, or apply disabled CSS styles consistently (`opacity: 0.5; cursor: not-allowed`).

### BUG-007: Users List Header Counts Mismatch Actual Data
**Location:** Users page header
**Impact:** Header reads "8 total / 0 banned" while the table shows 200 users including banned entries.
**Reproduced:** Yes (E2E screenshot `e2e-04-users-page.png`)
**Fix:** Derive the header counts from the same data source as the table.

### BUG-008: Duplicate MongoDB Index on `email` Field in Admin Model
**Location:** `apps/api/src/models/admin.model.ts`
**Impact:** MongoDB creates two indexes for the same field, wasting storage and slightly degrading write performance.
**Reproduced:** Check `db.admins.getIndexes()` — two indexes covering `email`.
**Fix:** Remove `index: true` from the field definition if `unique: true` is already present (unique implies an index).

### BUG-009: WebSocket Connection Repeatedly Failing
**Location:** Dashboard — browser console
**Impact:** `ws://localhost:5001` connection fails repeatedly in the browser (observed during E2E testing). Real-time metrics (if any depend on WebSocket) will not update.
**Note:** The live monitor is marked "Coming in Task 12" — these errors may be expected stubs. If WebSocket is not yet implemented, the client should not attempt to connect.

## P3 — Low

### UI-001: Forbidden Pages Show Content Gate Text Instead of Redirecting
**Example:** Auditor navigating to `/settings` sees a message "Settings are only accessible to Super Admins" but the URL still shows `/settings`.
**Impact:** Confusing UX — the browser URL bar shows a path the user cannot access.
**Fix:** Either redirect to `/` or the user's permitted landing page, or update the URL to reflect a canonical "forbidden" path.

### UI-002: Grammar — "1 reports" in Content Queue
**Location:** Content moderation queue item badges
**Impact:** "1 reports" should be "1 report" (singular).
**Fix:** Pluralise dynamically: `${count} report${count === 1 ? '' : 's'}`.

### UI-003: Live Monitor is a Stub Placeholder
**Location:** `/live-monitor`
**Impact:** Page shows "Coming in Task 12" — not a bug per se, but should be excluded from the nav until implemented to avoid dead-end navigation.

### UI-004: Audit Log Seed Data Contains Impossible RBAC Actions
**Location:** Audit log page
**Impact:** Seed data shows `support` role performing "Invite Admin" and `moderator` performing "Update Settings" — neither of which is permitted by the RBAC config. This could mislead QA or audit reviews.
**Fix:** Regenerate seed data that respects the ROLE_PERMISSIONS matrix.

### DATA-001: No `GET /api/v1/users/:id/videos` Endpoint
**Impact:** Profile video grid falls back to an empty state placeholder.

### SEC-016: Content-Type Not Enforced on Login (Form-Encoded Accepted)
**Endpoint:** `POST /api/v1/auth/login`
**Impact:** `application/x-www-form-urlencoded` bodies are silently rejected (body parses as empty). Not a current exploit, but inconsistent behavior could mask future issues.
**Fix:** Add `Content-Type: application/json` enforcement middleware, or respond 415 for non-JSON on JSON-only routes.

### SEC-017: `refresh_token` Cookie Path is `/api/v1/auth/refresh` Only
**Impact:** The refresh token cookie (`admin_refresh_token`) is scoped to `Path=/api/v1/auth/refresh`. This is correct and intentional, but means the cookie is invisible in browser dev-tools Network tab for other requests, which can confuse developers debugging auth.

## Informational

### INFO-001: CORS Origin Not Reflected (Correct Behavior Confirmed)
The API correctly returns `Access-Control-Allow-Origin: http://localhost:3000` for all origins — arbitrary origins are not reflected. This was explicitly tested with `Origin: http://evil.example.com` → no wildcard or echo.

### INFO-002: JWT `none` Algorithm Correctly Rejected
`alg: none` tokens are rejected with 401. Tested explicitly.

### INFO-003: Sensitive Data Not Exposed
- Login response: no `passwordHash` field ✓
- Admins list: no `passwordHash` field ✓
- Refresh token: HttpOnly cookie only — not in response body ✓

### INFO-004: Helmet.js Headers Fully Configured
All expected security headers present:
- `X-Content-Type-Options: nosniff` ✓
- `X-Frame-Options: SAMEORIGIN` ✓
- `Content-Security-Policy` (full policy) ✓
- `X-Powered-By` suppressed ✓
- `Strict-Transport-Security` ✓
- `Referrer-Policy: no-referrer` ✓
